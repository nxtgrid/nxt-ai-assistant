# Adding an MCP Server

This guide walks through adding a new MCP tool server from scratch — the files to create, where to register it, and how to define tools. It uses a hypothetical `weather` server as a running example.

## Concepts

- **MCP Server** — a Python module that exposes a set of tools Gemini can call during a conversation. Each server wires up `handle_list_tools()` and `handle_call_tool()` — generated by `ToolRegistry`, not hand-written (see below).
- **`ToolRegistry`** (`mcp_servers/shared_code/tool_registry.py`) — declares a tool's name, schema, and handler together at one `@registry.tool(...)` site, then generates `handle_list_tools`/`handle_call_tool` from the registrations. This replaced the old pattern of three hand-synced copies (a schema list, an `if/elif` dispatch chain, and per-branch result wrapping) that was the root cause of tool-manifest drift.
- **`tool_schemas.py`** — each server's `TOOL_SCHEMAS` list of plain dicts, one per tool. This is what you edit to add/change a tool's schema — not `tool_definitions.json` directly (see Step 3).
- **`tool_definitions.json`** — a generated snapshot of every server's tools, regenerated from code via `scripts/export_tools.py`. This is the manifest prod actually serves (`server_registry.list_tools` prefers it over live code), so it must be regenerated — never hand-edited — whenever a server's tools change. The one exception is `grafana`, whose manifest is computed at runtime from dashboard metadata and is deliberately excluded from export.
- **Server Registry** — `mcp_servers/server_registry.py` maps server names to their Python modules (`SERVER_METADATA`) and lazily imports them.

## Step 1 — Create the Server Module

Create `mcp_servers/servers/weather_server/tool_schemas.py`:

```python
"""Tool schemas for the Weather MCP server."""

from typing import Any, Dict, List

TOOL_SCHEMAS: List[Dict[str, Any]] = [
    {
        "name": "get_forecast",
        "description": "Get a weather forecast for a location.",
        "inputSchema": {
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "City name or coordinates"},
            },
            "required": ["location"],
        },
        "visible_to_customer": False,
    },
]
```

Create `mcp_servers/servers/weather_server/weather_mcp_server.py`:

```python
import asyncio
import logging
import sys

import mcp.types as types
from mcp.server import Server
from shared_code.stdio_runner import run_stdio_server
from shared_code.tool_registry import ToolRegistry

from .tool_schemas import TOOL_SCHEMAS

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(name)s] %(levelname)s: %(message)s",
    handlers=[logging.StreamHandler(sys.stderr)],
)
logger = logging.getLogger("weather-server")

server = Server("weather-server")
registry = ToolRegistry("weather")
_SCHEMAS_BY_NAME = {s["name"]: s for s in TOOL_SCHEMAS}


@registry.tool("get_forecast", _SCHEMAS_BY_NAME["get_forecast"])
async def _get_forecast(arguments: dict) -> list[types.TextContent]:
    location = arguments.get("location", "")
    # Replace with your actual API call
    from shared.utils.response_formatters import compose_json_response

    return list(compose_json_response({"location": location, "forecast": "sunny"}))


# Everything server_registry needs for in-process (prod) use:
handle_list_tools = registry.handle_list_tools
handle_call_tool = registry.handle_call_tool

# stdio mode (Claude Desktop / local dev) additionally needs these decorated:
handle_list_tools = server.list_tools()(handle_list_tools)
handle_call_tool = server.call_tool()(handle_call_tool)


async def main():
    await run_stdio_server(server, name="weather-server", label="Weather")


if __name__ == "__main__":
    asyncio.run(main())
```

Also create `mcp_servers/servers/weather_server/__init__.py` (empty file).

Notes:
- The string literal passed as the tool's `name` **must equal `schema["name"]`** — `ToolRegistry.tool()` raises `ValueError` at import time otherwise. This is deliberate: it's the static evidence `mcp_servers/tests/test_tool_manifest_sync.py` uses to prove every declared tool actually dispatches.
- Return values pass through untouched — return `TextContent`, or a mix of `TextContent`/`ImageContent` for charts/images. Don't raise from inside a `@registry.tool` handler for expected failure modes; use `compose_error_response`/`compose_json_response` (`shared/utils/response_formatters.py`) so the caller gets readable JSON instead of a sanitized generic error.
- Need a write-gated tool (hidden/refused unless `WEATHER_ACTIONS_ENABLED=true`)? Pass `gated=True` to `@registry.tool(...)`. Need a whole-server disabled-state gate checked before any tool dispatches? Use `@registry.pre_dispatch`.

## Step 2 — Register the Server

Add an entry to `SERVER_METADATA` in `mcp_servers/server_registry.py`:

```python
"weather": {
    "description": "Weather forecasts and current conditions",
    "module": "servers.weather_server.weather_mcp_server",
},
```

And the matching entry in `SERVER_CONFIGS` in `mcp_servers/handler.py` (the serverless/HTTP entry point — kept in sync with `server_registry.py` and test-enforced by `mcp_servers/tests/test_server_list_sync.py`):

```python
"weather": {
    "module": "servers.weather_server.weather_mcp_server",
    "description": "Weather forecasts and current conditions",
},
```

Add it to `MCP_SERVER_NAMES` in `shared/config/flag_registry.py` — this is what generates the `WEATHER_ENABLED` flag (test-enforced by `chat_orchestrator/tests/test_flag_registry.py`), then regenerate the example file:

```bash
python -m shared.config.flag_registry > shared/config/flags.env.example
```

If your server name contains underscores (e.g., `solar_alerts`), also add it to `multi_word_servers` in `chat_orchestrator/orchestrator/services/tool_executor.py` (line ~62):

```python
multi_word_servers = [
    "equipment_diagnostics",
    "equipment_control",
    "payment_processor",
    "grid_design",
    "solar_alerts",  # add here if your name has underscores
]
```

Single-word server names (like `weather`) don't need this.

## Step 3 — Regenerate `tool_definitions.json`

Once your `tool_schemas.py` and `@registry.tool(...)` handlers are in place, regenerate the manifest — **don't hand-edit `tool_definitions.json`**, it's a generated snapshot that would just drift again:

```bash
cd mcp_servers && ../chat_orchestrator/.venv/bin/python scripts/export_tools.py
```

Confirm code and JSON agree:

```bash
python -m shared.config.flag_registry | diff - shared/config/flags.env.example  # should be empty
```

### Gemini inputSchema constraints (critical)

- Property types must be `string`, `integer`, `number`, `boolean`, or `array` with `items`.
- **Never use `"type": "object"` without nested `"properties"`** — Gemini rejects it with a 400 error that breaks every staff conversation's tool payload.
- If you need to pass a dict/JSON blob, declare it as `"type": "string"` and parse it in the handler.

### Advertised vs. internal-only tools

A tool can exist and be callable (`server_registry.call_tool` always works) without being offered to the LLM. Set `"internal_only": True` in the tool's schema dict for tools meant to be called programmatically (e.g. by another tool's handler) rather than chosen by Gemini — `user_permissions._filter_and_convert_tools` drops these before the LLM ever sees them.

## Step 4 — Enable/Disable Controls

**Server-level:** Set `WEATHER_ENABLED=false` in the environment to hide all weather tools.

**Per-tool action gating:** Pass `gated=True` to `@registry.tool(...)` — the tool is hidden from `handle_list_tools` and refused by `handle_call_tool` whenever `WEATHER_ACTIONS_ENABLED` is false.

**Per-tool disable (any tool, any server):** Add `"weather:get_forecast"` to the `MCP_DISABLED_TOOLS` JSON array in env to disable one tool while keeping others enabled.

**Admin UI:** Settings → MCP Servers & Tools provides toggles without an environment variable change. Note `anansi_app/nicegui_app/pages/settings.py`'s `_MCP_SERVER_KEYS` frozenset is hand-maintained (it does not derive from `MCP_SERVER_NAMES`) — add your new `{SERVER}_ENABLED` key there too.

## Step 5 — Test Without the Full Stack

The `mcp_servers` venv has credentials for remote databases. `PYTHONPATH` must include both `mcp_servers/` and the repo root so `shared/` is importable:

```bash
cd mcp_servers && source .venv/bin/activate

PYTHONPATH=$PWD:.. python -c "
import asyncio
from servers.weather_server.weather_mcp_server import handle_call_tool

async def test():
    result = await handle_call_tool('get_forecast', {'location': 'Lagos'})
    for item in result:
        print(item.text)

asyncio.run(test())
"
```

Then run the test suites that guard this all staying in sync:

```bash
chat_orchestrator/.venv/bin/python -m pytest mcp_servers/tests/ -q
chat_orchestrator/.venv/bin/python -m pytest chat_orchestrator/tests/ -q
```

## Injected Arguments (Security Model)

The `tool_executor` injects context fields into every tool call's `arguments` dict before forwarding to the server. These fields are **not visible to Gemini** and cannot be spoofed by the LLM:

| Field | Description |
|-------|-------------|
| `chat_id` | Telegram chat ID of the originating message |
| `topic_id` | Telegram topic/thread ID (groups) |
| `user_email` | Resolved email of the requesting user |
| `organization_id` | Resolved org ID |
| `session_id` | Chat session UUID |

Use these to scope queries to the right user or organization. **Never trust user-provided org or user IDs from LLM arguments for access decisions.**
